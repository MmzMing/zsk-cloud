### 请求处理链路全景 (过滤器链)

#### 1. 链路执行顺序图

```text
客户端请求 
   │
[ 网关层 (zsk-gateway) ]
   │  1. XssFilter (请求清洗)
   │  2. Whitelist Check (白名单放行)
   │  3. AuthFilter (Token校验 -> Redis过期时间续期)
   │  4. Header Injection (用户信息透传)
   │  5. Sentinel (限流降级)
   │  6. LoadBalancer (负载均衡转发)
   ↓
[ 业务层 (微服务) ] ── 由 SecurityConfiguration 统一配置
   │  7. RepeatSubmitFilter (防重提交校验)
   │  8. HeaderContextFilter (Header转ThreadLocal上下文)
   │  9. AuthorizationFilter (Spring Security 权限判定)
   │      │
   │      ├─ 判定失败 -> 返回 401/403 JSON
   │      └─ 判定通过 -> 进入 Controller
   ↓
[ 业务处理 ]
      10. 执行业务代码 (通过 SecurityUtils 获取用户信息)
```

#### 2. 链路详细描写
##### 1. 网关层处理 (zsk-gateway)
网关作为统一入口，负责全局的安全与流量控制：
- **XSS 过滤 (XssFilter)**：清洗请求参数（Query/Body）中的恶意 HTML 标签与脚本，防止 XSS 攻击。
- **白名单校验**：检查请求路径是否匹配 `security.ignore.whites`。匹配则跳过后续 Token 校验，直接转发。
- **认证拦截 (AuthFilter)**：
    - 从请求头 `Authorization` 中解析 Token。
    - 校验 Token 在 Redis 中的有效性。
    - **Token 续期**：校验成功后，自动延长 Redis 中 Token 的有效期（实现滑动过期）。
- **用户信息透传**：将解析出的用户 ID、用户名、权限等信息封装进自定义请求头（如 `X-User-Id`），向下游微服务透传。
- **流量治理 (Sentinel)**：根据预设规则进行限流、熔断与降级保护。

##### 2. 业务层处理 (zsk-common-security)
业务层的安全逻辑由 SecurityConfiguration.java 统一配置和驱动：
- **防重提交 (RepeatSubmitFilter)**：拦截短时间内完全相同的请求，防止表单重复提交。
- **上下文恢复 (HeaderContextFilter)**：
    - 读取网关透传的 `X-User-Id` 等请求头。
    - 将用户信息存入 `SecurityContext` (基于 ThreadLocal)，使业务代码可通过工具类获取当前登录人。
- **权限判定 (AuthorizationFilter)**：
    - 结合 `isAuthenticated` 方法进行决策。
    - 验证用户是否已登录或请求是否来自受信任的内部调用（`from-source: inner`）。
- **异常响应**：
    - **401 (Unauthorized)**：认证失败时，返回 JSON 响应。**注意：** 后端不执行 `/login` 跳转，由前端拦截 401 状态码后主动跳转。
    - **403 (Forbidden)**：已登录但无权限访问特定资源时，返回 JSON 响应。
